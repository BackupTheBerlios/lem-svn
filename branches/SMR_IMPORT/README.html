Welcome to LEM - A text-based language for XTUML

What is LEM?

LEM is a Language for Executable Models. Read the sample model to get a feel for LEM and peruse the LEM grammar specification

The goal is to be able to express complete XTUML models in a plain text language. LEM defines XTUML models without the reliance on UML graphics (diagrams). The motivation for a plain text representation are:

  • plain text easier to understand when compared to UML diagrams
  • plain text can be searched by Google (and other text-based search engines)
  • version control is facilitated (try diff'ing UML diagrams)
  • models can be recorded and modified with a simple text editor
Plain text models are a power and expressive tool which encourage full engagement of the non-modelling domain experts during the modelling process.

Project Objectives

The objectives of this project is to
  1. define the grammar for LEM
  2. develop a parser for the language
  3. explore the use of LEM and the parser in the context of an open source XTUML toolkit

As a broad objective, an XTUML model expressed in LEM should be easy to read and covey significant concepts to all readers.

Status

The project is in its early stages. The initial goal is to establish a working language definition that covers the main elements of XTUML. Once the LEM definition is reasonably complete, and we have a parser, the language constructs will be reviewed by testing against real-world XTUML model.

The status so far is summaried in terms of XTUML elements

Done and working:

  1. Gross structures (Model, domains, subsystems)
  2. Core data types
  3. Domain types
  4. Class definition with class attributes
  5. Generalisation relationships
  6. Associations (with multiplicity and verb clause annotations)
  7. Association classes

The parser will parse an LEM language input file and build a JAVA object graph representing the model. Some aspects of static verification have been included but there is a lot to do.

Coming soon:

  1. State machine, states, events and transitions
  2. Action language (state procedures)
  3. OCL
  4. Static verification
